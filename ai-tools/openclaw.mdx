Set up OpenClaw Skills to help maintain this documentation.

## Prerequisites

- OpenClaw installed
- Your docs repository opened in a workspace

## Setup

<Steps>
<Step title="Install and verify OpenClaw">
  ```bash
  curl -fsSL https://openclaw.ai/install.sh | bash
  openclaw doctor
  openclaw status
  ```
</Step>

<Step title="Create a workspace skill">
  Create a local skill in your docs repo:

  ```bash
  mkdir -p skills/memex-docs
  ```

  Then add `skills/memex-docs/SKILL.md`:

  ```md
  ---
  name: memex-docs
  description: Keep Memex API docs accurate and publish validated changes.
  ---

  When editing docs in this repository:
  1. Update relevant `.mdx` and `api-reference/openapi.json` files.
  2. Run `npx -y mint@latest validate`.
  3. Run `npx -y mint@latest openapi-check api-reference/openapi.json` when OpenAPI changes.
  4. Commit and push to `main`.
  ```
</Step>

<Step title="Discover and validate skills">
  ```bash
  openclaw skills list
  openclaw skills list --eligible
  openclaw skills info memex-docs
  openclaw skills check
  ```
</Step>

<Step title="Restart session after skill changes">
  OpenClaw snapshots eligible skills at session start. After editing skills or
  `openclaw.json`, start a new session (or rely on watcher refresh) before
  testing.
</Step>
</Steps>

## Skill locations and precedence

OpenClaw loads skills from:

- Workspace `./skills` (highest precedence)
- Managed `~/.openclaw/skills`
- Bundled skills (lowest precedence)

If names conflict: workspace overrides managed, and managed overrides bundled.

## Optional: install skills from ClawHub

```bash
npm i -g clawhub
clawhub search "docs"
clawhub install <skill-slug>
```

## Per-skill config and secrets

Configure skill overrides in `~/.openclaw/openclaw.json` under
`skills.entries.<skillKey>`:

```json5
{
  skills: {
    entries: {
      "memex-docs": {
        enabled: true,
        env: {
          GITHUB_TOKEN: "${GITHUB_TOKEN}"
        }
      }
    }
  }
}
```

`skills.entries.*.env` and `skills.entries.*.apiKey` apply to host runs. For
sandboxed runs, set `agents.defaults.sandbox.docker.env` (or per-agent sandbox
env).

<Warning>
  Treat third-party skills as untrusted code. Review skill contents before
  enabling them.
</Warning>
