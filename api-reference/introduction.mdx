Memex exposes two API surfaces:

- **REST API** (canonical): direct HTTP endpoints.
- **MCP server**: tool interface that wraps the same backend handlers.

## Output shape parity

For each operation, Memex keeps the same response payload shape across:

- REST
- MCP (`result.structuredContent`)
- OpenClaw skills (when they return REST payloads unchanged)

See the canonical contract page:

- [Response shapes](/api-reference/response-shapes)

## Base URLs

- REST (public router): `https://api.memex.garden`
- REST (direct Supabase): `https://rubctkzqxcdgkbbcprnu.supabase.co/functions/v1`
- MCP server: `https://mcp.memex.garden/mcp`

## REST vs MCP

| Surface | Auth | Billing behavior | Backing handlers |
| --- | --- | --- | --- |
| REST | `Authorization: Bearer <api_key>` | No MCP credit deductions by default | `searchContentApi`, `saveContentByUrl` |
| MCP | `x-user-id` + `x-api-key` (or `userId` / `apiKey` query params), using the same key | `search_content`: `1` credit per request, plus `+1` credit when rerank is used | Same handlers, with MCP wrapper flags |

See [API Keys](/api-reference/api-keys) for where to create/find your key in Settings.

## Search limits

- `search-content` / `search_content` supports `limit` up to `500` results per call.

## Tool mapping

| MCP tool | Equivalent REST endpoint | Wrapper behavior |
| --- | --- | --- |
| `search_content` | `POST /search-content` | Forces `includeReferences=true` and `applyBilling=true` |
| `save_content_by_url` | `POST /save-content-by-url` | Passes through arguments unchanged |

  If you only need HTTP integration, use the REST API pages in this section. The MCP pages describe protocol/auth differences and billing semantics.
